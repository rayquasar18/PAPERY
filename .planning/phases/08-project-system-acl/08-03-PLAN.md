<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| owner→invite endpoints | Project administrators submit role-bearing collaboration requests and optional invitee emails |
| public token holder→accept endpoint | Invite token crosses into authenticated acceptance flow and must not be replayed or extended |
| owner→member mutation endpoints | Membership changes alter authorization state and can orphan a project if not guarded |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-09 | T | `backend/app/services/project_service.py` invite acceptance | mitigate | Persist hashed token, enforce `expires_at`, and atomically mark `accepted_at` so a token cannot be replayed |
| T-08-10 | I | `backend/app/api/v1/projects.py` invite/member endpoints | mitigate | Keep invite creation and member mutation behind owner-only ACL dependency; accept endpoint stays authenticated to bind membership to a known user |
| T-08-11 | E | `backend/app/repositories/project_member_repository.py` owner counts | mitigate | Count active owners before demotion/removal and reject operations that would leave zero owners |
| T-08-12 | R | Invite/member regressions | mitigate | Maintain separate `tests/test_project_invites.py` and `tests/test_project_members.py` suites covering expiry, replay, role fidelity, and final-owner rejection |
</threat_model>
