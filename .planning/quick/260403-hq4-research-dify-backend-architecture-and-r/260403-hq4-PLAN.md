# Quick Plan: Restructure PAPERY Backend (Dify-Aligned)

## Context

Research found PAPERY is already well-aligned with Dify patterns (extensions, config MI composition, models). Three structural changes needed before Phase 3 (Auth) starts:

1. Move `core/config/` → `configs/` (Dify's top-level config separation)
2. Delete empty `crud/` directory (adopt Dify's single `services/` layer)
3. Create `libs/` and `tasks/` scaffolds for future use

The refactor is low-risk — Phase 1-2 have minimal code, and the change is mostly import path updates.

---

## Tasks

### Task 1: Move `core/config/` → `configs/` and update all imports

**What:** Relocate the config module from `app/core/config/` to `app/configs/` to match Dify's pattern where `configs/` is a top-level concern, not nested inside `core/`. Update all import paths project-wide.

**Files to move:**
- `app/core/config/__init__.py` → `app/configs/__init__.py`
- `app/core/config/app.py` → `app/configs/app.py`
- `app/core/config/database.py` → `app/configs/database.py`
- `app/core/config/redis.py` → `app/configs/redis.py`
- `app/core/config/minio.py` → `app/configs/minio.py`
- `app/core/config/security.py` → `app/configs/security.py`
- `app/core/config/email.py` → `app/configs/email.py`
- `app/core/config/cors.py` → `app/configs/cors.py`
- `app/core/config/admin.py` → `app/configs/admin.py`

**Import updates required (13 files):**
- `app/main.py` — `from app.core.config import settings` → `from app.configs import settings`
- `app/extensions/ext_database.py` — same change
- `app/extensions/ext_redis.py` — same change
- `app/extensions/ext_minio.py` — same change
- `app/api/v1/health.py` — same change
- `migrations/env.py` — same change
- `app/configs/__init__.py` — update internal imports (`app.core.config.X` → `app.configs.X`)
- `tests/test_config.py` — update all imports (~8 occurrences)

**After move:** Delete empty `app/core/config/` directory. Keep `app/core/` with only `__init__.py` and `exceptions/`.

**Verify:** `cd backend && uv run python -m pytest tests/ -v`

### Task 2: Delete `crud/`, create `libs/` and `tasks/` scaffolds

**What:** Remove the empty `crud/` directory (Dify pattern: business logic + data access in `services/`, no separate CRUD layer). Create `libs/` for future shared utilities and `tasks/` for future background job definitions.

**Changes:**
1. **Delete** `app/crud/__init__.py` and `app/crud/` directory
2. **Create** `app/libs/__init__.py` with module docstring:
   ```python
   """Shared utility libraries (password, encryption, pagination, etc.)."""
   ```
3. **Create** `app/tasks/__init__.py` with module docstring:
   ```python
   """Background task definitions (ARQ/Celery workers)."""
   ```
4. **Update** `app/services/__init__.py` — add docstring clarifying it's the primary business logic layer:
   ```python
   """Business logic and data access layer. Services contain all domain logic."""
   ```

**Verify:** `cd backend && uv run python -m pytest tests/ -v`

## Decision Record

| Decision | Rationale |
|----------|-----------|
| `configs/` not `config/` | Matches Dify naming convention exactly |
| Keep `api/` not `controllers/` | FastAPI convention; Dify's `controllers/` is Flask-specific |
| Keep `middleware/` | ASGI middleware is first-class in FastAPI; Dify omits it because Flask uses different patterns |
| Keep `schemas/` | Pydantic schemas are superior to Marshmallow fields; our approach is cleaner |
| Delete `crud/` | Dify proves single `services/` layer works well; eliminates unnecessary abstraction |
| Keep `dependencies.py` | FastAPI's `Depends()` DI is the correct auth injection pattern |
| Static-method services pattern | Adopted from Dify — stateless, testable, no instance overhead |
