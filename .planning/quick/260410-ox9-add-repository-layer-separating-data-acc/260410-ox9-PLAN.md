# Quick Plan: Add Repository Layer Separating Data Access from Business Logic

**ID:** 260410-ox9
**Created:** 2026-04-10
**Status:** ready
**Branch:** feature/repository-layer

---

## Goal

Introduce an `app/repositories/` layer to separate data access (SQLAlchemy queries) from business logic (service orchestration). This replaces the current single `services/` approach where `auth_service.py` mixes DB queries with business rules.

## Context

- `auth_service.py` currently contains 2 pure data-access functions (`get_user_by_email`, `get_user_by_uuid`) mixed with 6 business logic functions
- 3 consumers import from `auth_service`: `api/v1/auth.py`, `api/dependencies.py`, `scripts/create_first_superuser.py`
- Decision log entry "Single services/ layer replaces crud/ + services split" — this task reverses that decision with a proper repository pattern
- The User model uses soft-delete (via `SoftDeleteMixin`), UUID+BigInt dual-ID, and has an OAuthAccount relationship

## Architecture Decision

```
API Layer  →  Service Layer  →  Repository Layer  →  SQLAlchemy / DB
(routes)      (business logic)   (data access)       (models)
```

- **Repository**: Pure data access — SELECT, INSERT, UPDATE, DELETE. No business rules, no exceptions beyond DB errors.
- **Service**: Business logic, validation, orchestration. Calls repository methods. Raises domain exceptions.

## Tasks

### Task 1: Create Base Repository + UserRepository

**Files to create:**
- `backend/app/repositories/__init__.py`
- `backend/app/repositories/base.py` — Generic `BaseRepository[ModelType]` with async CRUD ops
- `backend/app/repositories/user_repository.py` — `UserRepository(BaseRepository[User])` with user-specific queries

**BaseRepository[ModelType] methods:**
- `get_by_id(id: int) -> ModelType | None`
- `get_by_uuid(uuid: UUID) -> ModelType | None`
- `create(instance: ModelType) -> ModelType` — add + commit + refresh
- `update(instance: ModelType) -> ModelType` — commit + refresh
- `soft_delete(instance: ModelType) -> ModelType` — set deleted_at + commit

**UserRepository additional methods (extracted from auth_service):**
- `get_by_email(email: str) -> User | None` — case-insensitive, soft-delete-aware (from `get_user_by_email`)
- `get_active_by_uuid(uuid: UUID) -> User | None` — soft-delete-aware (from `get_user_by_uuid`)
- `create_user(email, hashed_password, is_active, is_verified, is_superuser) -> User`

**Design rules:**
- All methods take `AsyncSession` as constructor param (repo is instantiated per-request)
- BaseRepository filters `deleted_at.is_(None)` automatically for soft-delete models
- No business exceptions — return `None` when not found, let services interpret
- Type-safe with generics: `class BaseRepository(Generic[ModelType])`

### Task 2: Refactor auth_service.py to Use UserRepository

**Changes to `backend/app/services/auth_service.py`:**
- Remove `get_user_by_email()` and `get_user_by_uuid()` functions (moved to repository)
- All functions that take `db: AsyncSession` now also receive or construct a `UserRepository`
- `register_user()` → uses `user_repo.get_by_email()` + `user_repo.create()`
- `authenticate_user()` → uses `user_repo.get_by_email()`
- `rotate_refresh_token()` → uses `user_repo.get_active_by_uuid()`
- `verify_email()` → uses `user_repo.get_active_by_uuid()` + `user_repo.update()`
- `create_first_superuser()` → uses `user_repo.get_by_email()` + `user_repo.create()`

**Changes to consumers:**
- `api/v1/auth.py` — replace `auth_service.get_user_by_email(db, ...)` and `auth_service.get_user_by_uuid(db, ...)` with `UserRepository(db)` calls or updated service signatures
- `api/dependencies.py` — replace `auth_service.get_user_by_uuid(db, uuid)` with `UserRepository(db).get_active_by_uuid(uuid)` (this is a pure data lookup, repository is more appropriate)
- `scripts/create_first_superuser.py` — update import if needed

**Pattern for service functions:** Accept `UserRepository` as parameter (dependency injection):
```python
async def register_user(user_repo: UserRepository, email: str, password: str) -> User:
```

OR instantiate inside service functions (simpler, fewer caller changes):
```python
async def register_user(db: AsyncSession, email: str, password: str) -> User:
    user_repo = UserRepository(db)
    ...
```

**Decision:** Use the second pattern (instantiate inside) for v1 — minimizes API surface changes. The `db` parameter stays, callers unchanged. Dependencies (`dependencies.py`) that do pure lookups can use `UserRepository` directly.
