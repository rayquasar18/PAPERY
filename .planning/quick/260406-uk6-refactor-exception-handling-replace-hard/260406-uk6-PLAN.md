# Quick Task: Refactor Exception Handling

**ID:** 260406-uk6
**Type:** refactor
**Branch:** `develop`
**Scope:** Move exception logic out of `main.py` into `core/exceptions/`, introduce `PaperyHTTPException`

---

## Context

`main.py` currently contains ~50 lines of exception handling logic including a hardcoded `error_code_map` dict. This violates the principle that `main.py` should be a clean entry point — wiring only, no business logic. The `core/exceptions/__init__.py` is essentially empty (just a docstring saying "use FastAPI HTTPException directly"). This refactor creates a proper custom exception class and moves all handler functions to a dedicated module.

## Tasks

### Task 1: Create `PaperyHTTPException` and convenience subclasses

**File:** `backend/app/core/exceptions/__init__.py`

Create `PaperyHTTPException(HTTPException)` with an `error_code: str` attribute. This is the exception endpoints will raise. Also define common convenience subclasses for readability.

```python
# PaperyHTTPException — adds error_code to FastAPI's HTTPException
class PaperyHTTPException(HTTPException):
    def __init__(
        self,
        status_code: int,
        error_code: str,
        detail: str = "An error occurred",
        headers: dict[str, str] | None = None,
    ) -> None:
        super().__init__(status_code=status_code, detail=detail, headers=headers)
        self.error_code = error_code

# Convenience subclasses (each sets status_code + error_code defaults):
# - BadRequestError(400, "BAD_REQUEST")
# - UnauthorizedError(401, "UNAUTHORIZED")
# - ForbiddenError(403, "FORBIDDEN")
# - NotFoundError(404, "NOT_FOUND")
# - ConflictError(409, "CONFLICT")
# - RateLimitedError(429, "RATE_LIMITED")
# - InternalError(500, "INTERNAL_ERROR")
```

Also define `HTTP_STATUS_ERROR_CODE_MAP` as a module-level constant for fallback mapping (moved from main.py).

### Task 2: Create exception handlers module

**File:** `backend/app/core/exceptions/handlers.py`

Move all three exception handler functions from `main.py` here:

1. `http_exception_handler` — checks if exc is `PaperyHTTPException` → use `exc.error_code`; otherwise falls back to `HTTP_STATUS_ERROR_CODE_MAP`
2. `validation_error_handler` — unchanged logic, just moved
3. `unhandled_exception_handler` — unchanged logic, just moved
4. `_get_request_id()` helper — moved here (private to this module)
5. `register_exception_handlers(app: FastAPI) -> None` — wiring function that main.py calls

**Backward compatibility:** If someone raises plain `HTTPException(status_code=404, detail="...")`, the handler still works via the fallback map. If they raise `PaperyHTTPException(status_code=404, error_code="USER_NOT_FOUND", detail="...")`, the custom error_code is used directly.

### Task 3: Slim down `main.py` and update tests

**File:** `backend/app/main.py`

Replace the ~50 lines of exception handling with a single import + call:

```python
from app.core.exceptions.handlers import register_exception_handlers
# ... after app creation ...
register_exception_handlers(app)
```

Remove: `_get_request_id`, `http_exception_handler`, `validation_error_handler`, `unhandled_exception_handler`, `error_code_map`, and now-unused imports (`RequestValidationError`, `StarletteHTTPException`, `JSONResponse`, `ErrorResponse`).

**File:** `backend/tests/test_exceptions.py`

Update tests to cover:
- `PaperyHTTPException` construction (status_code, error_code, detail, headers)
- Convenience subclass defaults (e.g., `NotFoundError("msg")` → 404 + "NOT_FOUND")
- `PaperyHTTPException` is instance of `HTTPException` (backward compat)
- Integration: raising `PaperyHTTPException` returns correct `error_code` in response
- Integration: raising plain `HTTPException` still returns mapped `error_code` (fallback)

Existing integration tests in `test_health.py` (lines 99-119) should pass unchanged — they test the response format, not the internal implementation.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| `PaperyHTTPException` inherits `HTTPException` | Yes | FastAPI catches HTTPException natively; our class IS-A HTTPException, just with extra `error_code` |
| Convenience subclasses | Yes, 7 common ones | `raise NotFoundError("User not found")` is cleaner than `raise PaperyHTTPException(404, "NOT_FOUND", "User not found")` |
| `register_exception_handlers()` pattern | Function in handlers.py | main.py stays clean; single call to register all handlers |
| Fallback map kept | Yes, in `__init__.py` | Plain `HTTPException` from middleware/deps still gets proper error_code |
