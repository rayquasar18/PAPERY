Read and analyze the following files for security, correctness, and best-practice issues:

1. `frontend/src/lib/api/client.ts` — Axios instance with 401 auto-refresh
2. `frontend/src/lib/api/auth.ts` — Typed auth API wrappers
3. `frontend/src/hooks/use-auth.ts` — TanStack Query auth hook (at new location after Task 1)
4. `frontend/src/proxy.ts` — Next.js proxy (i18n only, no auth guard)
5. `frontend/src/schemas/auth.ts` — Zod validation schemas (at new location after Task 1)
6. `frontend/src/lib/api/error.ts` — Error normalization

**Audit checklist (evaluate each and document findings):**

A. **Token refresh race condition:** Does the 401 interceptor queue correctly handle concurrent failing requests? Check: Are replayed requests using the new token or old? (Current pattern replays via `apiClient(originalRequest)` which should pick up new cookies automatically via `withCredentials: true` — verify this is correct.)

B. **Redirect loop risk:** If `/auth/refresh` itself returns 401, does `_retry` flag prevent infinite loop? (Current: yes, because `originalRequest._retry = true` is set before the refresh call, so the refresh 401 won't trigger another refresh.)

C. **Login redirect path:** `window.location.href = '/login'` does NOT use the i18n-aware router. After refresh failure, the user hits `/login` which the i18n middleware redirects to `/en/login` or `/vi/login`. This works but causes a double redirect. Should use locale-aware path.

D. **SSR safety:** The `window.location.href` check uses `typeof window !== 'undefined'`. Good. But the Axios instance is created at module scope — verify it won't execute interceptors during SSR in unexpected ways.

E. **CORS / withCredentials:** `withCredentials: true` is set. Backend CORS must include `credentials: true` and specific origin (not `*`). Document this as a requirement.

F. **Schema validation gaps:** `loginSchema` validates `password.min(8)` but no complexity rules (uppercase, lowercase, number). The `registerSchema` error message mentions complexity requirements but the regex doesn't enforce them. Flag as inconsistency.

G. **Error handling in mutations:** `useAuth` casts errors as `(err as { response?: ... })` — fragile. Should use the existing `normalizeError()` from `lib/api/error.ts` for consistent error handling.

H. **Proxy auth guard:** `proxy.ts` currently has NO auth guard — it only handles i18n. Document that protected routes (dashboard, settings) are accessible without authentication at the proxy level. Note this is by design (deferred to future plan) but flag as important to address.

I. **Query cache security:** `queryClient.clear()` on logout is correct — ensures no stale user data leaks between sessions.

J. **OAuth redirect URLs:** `googleLogin()` and `githubLogin()` use `window.location.href` to redirect to backend OAuth endpoints. These won't work during SSR. Should be guarded or only used in client components (which they currently are via `useAuth`).

Write findings to `AUTH-AUDIT.md` in the quick plan directory with:

- Severity: INFO / LOW / MEDIUM / HIGH
- Status: OK / NEEDS-FIX / DEFERRED
- Recommended fix (if applicable)
